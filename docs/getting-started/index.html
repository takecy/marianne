Getting Started

This guide walks you through setting up a local development environment for Marianne. By the end you’ll have the Tauri desktop app running, the test suite passing, and the documentation site previewing locally.

The Tauri Rust backend is intentionally minimal — src-tauri/src/lib.rs exposes only OS-integration commands (take_pending_open_paths, confirm_quit, renderer_ready, pick_and_open_image). Almost all feature development stays inside src/.

First-time setup

Section titled “First-time setup”

  1. Clone and install dependencies

  2. Install Playwright Chromium (for documentation Mermaid SSR)

    Note

    The architecture diagrams in this docs site are rendered to SVG at build time by a headless Chromium. You only need to run this once per machine — the browser is cached in ~/Library/Caches/ms-playwright (macOS) and reused on every pnpm docs:build.

  3. Launch the desktop app with hot reload

    The Vite dev server starts on http://localhost:1420. Tauri wraps it in a native window automatically.

Verification commands

Section titled “Verification commands”

Run these locally before every push — CI runs the same set, in order, and stops at the first failure.

CommandWhat it does
pnpm typecheckTypeScript strict check (tsc --noEmit)
pnpm lintESLint v10 flat config
pnpm fmt:checkdeno fmt --check for all source + Markdown
pnpm test:runVitest, single run
pnpm buildVite build of the React frontend
pnpm docs:buildBuilds this docs site into /docs/

Worktree-based workflow

Section titled “Worktree-based workflow”

For any non-trivial change, work in a dedicated worktree under .worktrees/:

.worktrees/ is gitignored. This keeps your main checkout clean and lets you run multiple parallel branches without git stash gymnastics.

Editing this docs site

Section titled “Editing this docs site”

Build before you commit

The deploy model is GitHub Pages → Branch: main / Folder: /docs. CI rejects any PR where the committed /docs/ is out of sync with pnpm docs:build output. Always run pnpm docs:build and commit the result alongside your site/ source edits.

Submitting a pull request

Section titled “Submitting a pull request”
  1. Create a feature branch in a worktree (see above).
  2. Make changes; run all verification commands.
  3. Commit using Conventional Commits in English (e.g. feat(canvas): add arrow rotation).
  4. Push and open a PR in draft mode by default.

Releases are fully automated via .github/workflows/tagging-release.yaml — see the repository README for the maintainer-side procedure.
