# Lessons

セッションを跨いで持ち越すべき学び。コード/設計の細部はリポジトリを読めば分かるので、ここには再現したくない失敗パターンと、その背景・対処を残す。

## 実装計画の値リテラル検証

- **症状**: Explore エージェントの調査要約だけを根拠にプランを書くと、判別共用体の discriminator 文字列（例: `"rect"` を `"rectangle"` と取り違える）や、コンポーネントの prop 名（例: `onColorChange` を `onActiveColorChange` と書き間違える）といった **リテラル値のミスマッチ** が紛れ込みやすい。型は付かないので人間レビューでも見落としやすく、実装時に初めて TypeScript エラーや回帰として現れる。
- **対処**: プランに具体的な識別子（discriminator・prop 名・interface 名・関数名）を書く前に、必ず `Read` か `grep` で実コードに当てて 1 回確認する。Explore の要約は「概念マップ」として扱い、リテラル値は実コードから直接引用する。
- **検出経路**: Codex レビュー (`/codex-review HIGH`) を実装計画に対して実行すると、HIGH 段階でこの種のミスマッチを高確率で拾ってくれる。

## 既存ハンドラの副作用を計画段階で全展開する

- **症状**: 「`handleColorChange` を wrapper に置き換える」のような計画は一見シンプルだが、既存ハンドラが想定外の副作用（例: `saveLastSelectedColor` での永続化）を持っている場合、wrapper 置換でその副作用が落ちて回帰する。
- **対処**: 既存ハンドラを差し替える/拡張する計画では、当該関数本体を Read で展開し、すべての行（呼び出している関数 / 更新している state）を計画文に列挙してから差分を提案する。「現状コード → 変更後コード」を並べて書くと、副作用の欠落が視覚的に検出できる。

## `patchByType` の参照同値性は「マッチ無し時のみ」

- **症状**: `canvasStore.patchByType` は id/type が一致したシェイプにパッチを当てると、たとえ **パッチ内容が現状と同じでも** `changed = true` にして新配列を返す。「同色再クリック → 履歴非汚染」を期待するなら、`patchByType` の参照同値性に頼るのは誤り。
- **対処**: 同値判定を含めた no-op 化が必要な場合、`patchByType` の手前（呼び出し側のアクション内）で `shape.color === color` のような値比較で早期 return する。判別共用体の narrowing が効くので、`switch (shape.type)` のケースごとに比較すれば mosaic のような color プロパティを持たない型も安全に弾ける。
- **応用**: 既存の `updateRect / updateText / updateArrow / updateMosaic` も同じ性質を持つ。値が変わらない部分更新でも履歴に積まれることに注意。
